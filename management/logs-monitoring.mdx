Observability is built into every VenixCloud service from the moment it is deployed. Whether you are tracking down a crash, validating a new feature, or watching resource consumption trend upward, the platform gives you real-time log streams and historical metrics without any additional instrumentation required on your part.

## Real-Time Logs

VenixCloud captures all output from your service and makes it available as a live stream in the dashboard.

<Steps>
  <Step title="Open the service">
    Go to **Dashboard → Services** and click the service you want to inspect.
  </Step>
  <Step title="Open the Logs tab">
    Select the **Logs** tab inside the service detail view. The log stream begins tailing automatically.
  </Step>
  <Step title="Read the output">
    Log lines appear in chronological order, newest at the bottom. Each line includes a timestamp, severity level, and message.
  </Step>
</Steps>

VenixCloud collects three categories of log output:

| Type | Source | Example |
|---|---|---|
| **stdout** | Normal application output written to standard out | `Server listening on port 3000` |
| **stderr** | Error and warning output written to standard error | `UnhandledPromiseRejectionWarning: TypeError` |
| **System events** | Platform-level events generated by VenixCloud itself | `Service restarted after OOM signal` |

<Tip>
  Use the **Level** filter above the log stream to show only `error` or `warn` lines when debugging. This cuts through high-volume `info` output and surfaces problems faster.
</Tip>

## Log Retention

VenixCloud retains logs for a rolling window that depends on your plan.

| Plan | Retention Period |
|---|---|
| **Free** | 7 days |
| **Pro** | 30 days |
| **Pro+** | 90 days |
| **Enterprise** | Custom (configurable up to 1 year) |

Logs older than the retention window are permanently deleted. If you need longer-term log archiving, use the API to stream logs to an external storage provider such as an S3-compatible bucket.

## Fetching Logs via API

You can retrieve log output programmatically for integration with external logging tools or automated analysis.

```bash
curl -X GET https://api.venixcloud.com/v1/services/{service_id}/logs \
  -H "Authorization: Bearer YOUR_API_KEY"
```

The API returns a JSON array of log entries. Each entry contains a timestamp, severity level, and the raw message string.

```json
[
  {
    "timestamp": "2025-01-15T10:23:41.882Z",
    "level": "info",
    "message": "Server listening on port 3000"
  },
  {
    "timestamp": "2025-01-15T10:23:42.104Z",
    "level": "info",
    "message": "Connected to database successfully"
  },
  {
    "timestamp": "2025-01-15T10:24:05.317Z",
    "level": "warn",
    "message": "Response time exceeded 2000ms for GET /api/users"
  },
  {
    "timestamp": "2025-01-15T10:24:19.558Z",
    "level": "error",
    "message": "Unhandled exception: Cannot read properties of undefined (reading 'id')"
  }
]
```

You can pass optional query parameters to scope the results:

| Parameter | Type | Description |
|---|---|---|
| `since` | ISO 8601 timestamp | Return only log lines after this time. |
| `until` | ISO 8601 timestamp | Return only log lines before this time. |
| `level` | string | Filter by severity: `info`, `warn`, or `error`. |
| `limit` | integer | Maximum number of lines to return (default `100`, max `1000`). |

## Resource Monitoring

Every running service exposes live and historical resource graphs under the **Metrics** tab in the service detail view.

<Card title="CPU Usage" icon="microchip">
  Displays percentage utilization of the allocated CPU over time. Sustained usage above 80% is a signal to scale up your plan or optimize your application.
</Card>

<Card title="RAM Usage" icon="memory">
  Shows current and peak memory consumption against your plan's allocation. Services that exceed their memory limit are automatically restarted.
</Card>

<Card title="Network I/O" icon="network-wired">
  Tracks inbound and outbound bytes per second. Useful for identifying unexpected traffic spikes or data-intensive operations.
</Card>

Metrics are displayed at 30-second resolution for the last hour and roll up to 5-minute averages for the previous 7 days. You can hover any point on a graph to see the exact value at that moment.

## Alerts

VenixCloud can notify you automatically when something goes wrong with your services.

<Note>
  Alerts are available on **Pro+ and Enterprise** plans. Free and Pro users can monitor services manually through the dashboard and log stream.
</Note>

The following alert conditions are supported:

- **Service down** — fires when a service stops responding to health checks for more than 60 seconds.
- **High memory** — fires when RAM usage exceeds a configurable threshold (default 90% of allocation) for more than 5 minutes.
- **Deployment failed** — fires whenever a deployment ends in the **Failed** state.

To configure alerts, open **Dashboard → Services → [Your Service] → Settings → Alerts**, add a notification channel (email, Slack webhook, or webhook URL), and set your thresholds.
