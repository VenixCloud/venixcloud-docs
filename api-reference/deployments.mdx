The Deployments API gives you visibility into every build and release event across all your VenixCloud services. Each time a service is updated, a new deployment record is created that tracks the full lifecycle from queue to live production. You can list all deployments, drill into a single deployment to see its status and logs, or trigger a new deployment programmatically — for example, as part of a CI/CD pipeline. All requests require a valid API key in the `Authorization` header.

---

## List All Deployments

Retrieve a list of deployments across all services on your account. Use the optional `service_id` query parameter to filter results to a specific service.

<CodeGroup>

```bash cURL — All Deployments
curl -X GET https://api.venixcloud.com/v1/deployments \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```bash cURL — Filtered by Service
curl -X GET "https://api.venixcloud.com/v1/deployments?service_id=svc_abc123" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

</CodeGroup>

### Query Parameters

<ParamField query="service_id" type="string">
  When provided, limits the response to deployments belonging to the specified service. If omitted, deployments from all services are returned sorted by `started_at` descending.
</ParamField>

### Response

**Status: `200 OK`**

```json
{
  "data": [
    {
      "id": "dep_m3w6v1",
      "service_id": "svc_abc123",
      "status": "running",
      "started_at": "2024-03-18T12:05:00Z",
      "finished_at": "2024-03-18T12:07:34Z"
    },
    {
      "id": "dep_q8e2k5",
      "service_id": "svc_abc123",
      "status": "failed",
      "started_at": "2024-03-17T09:30:00Z",
      "finished_at": "2024-03-17T09:31:15Z"
    }
  ],
  "total": 2
}
```

<ResponseField name="data" type="array">
  An array of deployment objects.

  <Expandable title="Deployment object fields">
    <ResponseField name="id" type="string">
      The unique identifier for the deployment, prefixed with `dep_`.
    </ResponseField>
    <ResponseField name="service_id" type="string">
      The ID of the service this deployment belongs to.
    </ResponseField>
    <ResponseField name="status" type="string">
      The current state of the deployment. One of `queued`, `building`, `deploying`, `running`, or `failed`.
    </ResponseField>
    <ResponseField name="started_at" type="string">
      ISO 8601 timestamp of when the deployment began processing.
    </ResponseField>
    <ResponseField name="finished_at" type="string">
      ISO 8601 timestamp of when the deployment completed, either successfully or with a failure. `null` if the deployment is still in progress.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="total" type="integer">
  The total number of deployments returned.
</ResponseField>

## Get a Deployment

Retrieve the full details of a single deployment by its ID. This endpoint returns the complete deployment record including a link to the live build logs, making it ideal for monitoring a deployment in progress or diagnosing a failed one.

<CodeGroup>

```bash cURL
curl -X GET https://api.venixcloud.com/v1/deployments/dep_m3w6v1 \
  -H "Authorization: Bearer YOUR_API_KEY"
```

</CodeGroup>

### Path Parameters

<ParamField path="id" type="string" required>
  The unique ID of the deployment to retrieve. Deployment IDs are prefixed with `dep_`.
</ParamField>

### Response

**Status: `200 OK`**

```json
{
  "id": "dep_m3w6v1",
  "service_id": "svc_abc123",
  "status": "running",
  "logs_url": "https://api.venixcloud.com/v1/deployments/dep_m3w6v1/logs",
  "started_at": "2024-03-18T12:05:00Z",
  "finished_at": "2024-03-18T12:07:34Z"
}
```

<ResponseField name="id" type="string">
  The unique identifier for this deployment.
</ResponseField>
<ResponseField name="service_id" type="string">
  The ID of the service associated with this deployment.
</ResponseField>
<ResponseField name="status" type="string">
  The current lifecycle status of the deployment. Possible values are:

  - `queued` — The deployment has been accepted and is waiting for a build runner.
  - `building` — Dependencies are being installed and the build command is running.
  - `deploying` — The built artifacts are being promoted to the live environment.
  - `running` — The deployment completed successfully and the service is live.
  - `failed` — The deployment encountered an error. Check `logs_url` for details.
</ResponseField>
<ResponseField name="logs_url" type="string">
  A URL to the streaming build and deployment logs for this deployment. Use this to inspect output from your build command or diagnose failures.
</ResponseField>
<ResponseField name="started_at" type="string">
  ISO 8601 timestamp of when the deployment entered the `building` state.
</ResponseField>
<ResponseField name="finished_at" type="string">
  ISO 8601 timestamp of when the deployment reached a terminal state (`running` or `failed`). Returns `null` for in-progress deployments.
</ResponseField>

## Trigger a Deployment

Manually trigger a new deployment for an existing service. This is useful when you want to redeploy without pushing a new commit — for example, to pick up updated environment variables, apply a configuration change, or re-run a build after a transient infrastructure issue.

<Note>
  This endpoint triggers a deployment using the same source code and configuration as the most recent previous deployment. To deploy new code, push a commit to the connected repository instead.
</Note>

<CodeGroup>

```bash cURL
curl -X POST https://api.venixcloud.com/v1/services/svc_abc123/deploy \
  -H "Authorization: Bearer YOUR_API_KEY"
```

</CodeGroup>

### Path Parameters

<ParamField path="service_id" type="string" required>
  The unique ID of the service to deploy. Service IDs are prefixed with `svc_`.
</ParamField>

### Response

**Status: `201 Created`**

```json
{
  "id": "dep_z1u4y7",
  "service_id": "svc_abc123",
  "status": "queued",
  "logs_url": "https://api.venixcloud.com/v1/deployments/dep_z1u4y7/logs",
  "started_at": null,
  "finished_at": null
}
```

<ResponseField name="id" type="string">
  The unique identifier for the newly triggered deployment.
</ResponseField>
<ResponseField name="service_id" type="string">
  The ID of the service being deployed.
</ResponseField>
<ResponseField name="status" type="string">
  Always `queued` immediately after the deployment is triggered. Poll `GET /deployments/:id` to track progress.
</ResponseField>
<ResponseField name="logs_url" type="string">
  The URL where build logs will be available once the deployment moves past the `queued` state.
</ResponseField>
<ResponseField name="started_at" type="string | null">
  `null` while the deployment is queued. Updated once a build runner picks it up.
</ResponseField>
<ResponseField name="finished_at" type="string | null">
  `null` until the deployment reaches a terminal state.
</ResponseField>

<Tip>
  After triggering a deployment, use `GET /deployments/{id}` on the returned `id` to poll for status changes, or follow the `logs_url` to stream real-time build output.
</Tip>
